# IncreaseX Architecture

## Overview

`increasex` is a Go wrapper around the Increase API that exposes the same product capabilities through two local interfaces:

- a human-oriented CLI
- a local MCP server over stdio for agent hosts such as Codex

The core architectural goal is that both interfaces behave the same way. Authentication, validation, normalization, preview logic, confirmation handling, idempotency, masking, and execution all flow through one shared application layer rather than being reimplemented separately for CLI and MCP.

## What The Product Does

At a high level, the project turns raw Increase API operations into safer, more ergonomic workflows for engineers and agents. Today that includes:

- authentication and local profile management
- account, account number, balance, transaction, card, document, event, external account, and transfer workflows
- preview-first write operations for money movement and object creation/update flows
- transfer approval queue actions
- interactive terminal navigation for common CLI paths
- a narrowly scoped MCP tool surface for local agent use

This is intentionally not a hosted control plane, not a browser app, and not a replacement for the official Increase CLI. It is a local operator and agent interface built directly on the official Increase Go client.

## Architectural Shape

The codebase is organized as a thin-interface, shared-core system:

- `internal/cli`
  Implements Cobra commands, interactive menus, and human-facing output behavior. It should stay thin and delegate business logic to the app layer.
- `internal/mcp`
  Implements a local JSON-RPC/MCP server over stdio. It defines tool schemas, dispatches tool calls, and returns structured envelopes, but relies on the same app services as the CLI.
- `internal/app`
  The heart of the system. This package owns the shared service surface, input validation, normalization into repo-defined summary/detail types, preview/execution flows, transfer rail handling, and confirmation-token verification.
- `internal/increase`
  A wrapper around the official Increase Go client. This isolates API calls, request-id capture, idempotency headers, a few custom endpoints, and error translation into the repo’s internal error model.
- `internal/auth` and `internal/config`
  Resolve credentials and profile state across flags, environment variables, durable config files, and macOS Keychain.
- `internal/util`
  Shared helpers for masking, money and time formatting, idempotency, and response envelopes.
- `internal/ui`
  Prompting helpers for the CLI’s interactive flows.

In practice, the dependency direction is:

`CLI / MCP -> app services -> Increase client wrapper / auth / config / util`

That direction matters because it keeps product behavior centralized. The CLI and MCP layers differ mainly in transport and presentation, not in operational semantics.

## Request Flow

Most operations follow the same pipeline:

1. The CLI command or MCP tool gathers user input and transport-specific options.
2. The interface resolves a session through `app.Services`, which in turn resolves credentials and environment through `internal/auth`.
3. The app layer validates and normalizes input into stable repo-defined types.
4. The app layer calls the `internal/increase` client wrapper, which talks to the official Increase API and captures request metadata such as request IDs.
5. The app layer normalizes raw Increase objects into consistent summaries/details used by both interfaces.
6. The interface renders the result for humans or returns a structured MCP response envelope.

The important design point is that normalization happens once in the shared core. That gives the CLI and MCP the same object shapes, error model, and write semantics.

## Preview-First Writes

All write paths are designed to be preview-first. Instead of immediately creating or mutating financial objects, the app layer first produces a preview result containing:

- a human-readable summary of the pending action
- a normalized detail payload
- a short-lived confirmation token

The confirmation token is generated by `internal/app/confirmation_service.go`. It is an HMAC-signed payload bound to:

- the operation name
- the normalized payload hash
- the selected profile
- the target environment
- a short TTL

Execution methods verify that token before making the real Increase API call. This design was chosen for two reasons:

- safety: accidental writes, especially money movement, require an explicit second step
- parity: CLI confirmations and MCP confirmations use the same primitive instead of inventing separate approval systems

For MCP specifically, this preserves a stateless tool model except for short-lived confirmation tokens, which keeps tool calls simple while still enabling safe writes.

## Authentication Model

Authentication is designed for local durability without forcing shell-only workflows. Credentials can resolve from:

1. explicit flags
2. environment variables
3. a durable local credentials file
4. a macOS Keychain mirror

The default login path stores a durable file credential and mirrors it to Keychain when available. We do not make Keychain the only default store because agent-launched or MCP-launched local processes cannot always rely on the same interactive Keychain access path as a human shell session. Session-only export still exists, but the architecture is optimized for local agent and CLI reuse across sessions.

## Response and Data Model

The repo does not expose raw Increase SDK objects directly at the product boundary. The app layer converts them into local summary/detail structs such as `AccountSummary`, `TransactionSummary`, `CardSummary`, `TransferDetails`, and `PreviewResult`.

This normalization gives the product a stable surface with a few benefits:

- CLI and MCP can share output contracts
- field naming and shaping stay consistent even if the upstream SDK evolves
- masking and derived fields can be applied centrally
- interfaces can remain task-oriented instead of mirroring raw API resources one-for-one

The shared response envelope also carries Increase request IDs, which is useful for debugging and support without making each transport invent its own result wrapper.

## Masking and Sensitive Data

Sensitive fields are masked by default. The product distinguishes between normal retrieval flows and explicitly sensitive retrieval flows, for example when returning account number or card details. This reduces the chance that normal browsing, listing, or agent-driven discovery leaks secrets unnecessarily.

That choice is especially important for MCP because agent tooling tends to compose operations automatically. Default masking makes the safe path the default path, while still allowing intentional access to sensitive details through dedicated operations.

## MCP Design Choices

The MCP server is intentionally local and stdio-only. The main reason is straightforward: we did not want to build, host, secure, and operate a remote MCP service. A local binary keeps deployment simple and fits the common agent-host model where a tool provider is registered as a child process on the user’s machine.

Once that choice is made, local-process behavior becomes part of the design surface. Things like environment inheritance, TTY availability, and Keychain access can differ from a normal interactive shell session, which is why the auth model is designed around a durable local file credential with Keychain as a mirror rather than the only store.

The MCP surface is deliberately narrow and task-oriented rather than exposing the full Increase API mechanically. Examples include tools like `list_accounts`, `get_balance`, `list_transfer_queue`, and `create_ach_transfer`. This was chosen to make agent behavior more reliable:

- schemas are easier to validate
- tools are easier to compose
- fewer low-level knobs reduce accidental misuse
- tool names align with concrete operator tasks rather than raw transport details

The repo also tracks CLI/MCP parity explicitly in [`docs/parity_matrix.md`](/Users/jessevaughan/Projects/Increase_CLI_wrapper_MCP/docs/parity_matrix.md), reinforcing that the same core workflow should exist in both surfaces.

## Error Handling

`internal/increase` translates SDK, network, and API errors into a repo-defined error model with stable codes such as authentication, validation, idempotency conflict, rate limiting, confirmation failure, and not found. That allows the CLI and MCP to present the same semantics even though one is a terminal UI and the other is a tool protocol.

This separation is important because upstream transport details are not always the right abstraction for users or agents. The product wants a small, predictable set of failure categories that map to operator actions.

## Why These Choices

The major design choices all support the same product goals:

- One shared app core: prevents drift between CLI and MCP behavior and keeps new features cheaper to add.
- Direct use of the official Increase Go client: reduces indirection and avoids shelling out to another CLI, which would be harder to type, test, and normalize.
- Preview-first writes: optimizes for safety in financial workflows.
- Narrow MCP tool schemas: improves reliability for agent callers.
- Local durable auth: makes CLI and MCP practical across sessions.
- Masked-by-default responses: protects sensitive financial data in routine usage.
- Local stdio MCP transport: avoids running a hosted service and stays easy to register with local agent hosts.

## How To Think About Extending It

A new feature should usually be added in this order:

1. add or extend the Increase client wrapper if the API call is new
2. add shared app-layer input types, validation, normalization, and preview/execution methods
3. expose the feature through CLI and/or MCP as thin adapters
4. preserve preview-first semantics for writes and masked-by-default semantics for sensitive data

That sequence keeps the architecture consistent: business behavior belongs in the shared core, while CLI and MCP remain transport layers over the same product logic.
